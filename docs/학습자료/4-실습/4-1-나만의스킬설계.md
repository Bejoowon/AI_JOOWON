# 나만의 스킬 설계

## ✏️ 설계 시간!

이메일 초안 스킬은 연습이었습니다. 이제 **진짜 나에게 필요한 스킬**을 설계해봅시다.

설계가 잘 되면 구현은 5분이면 됩니다.

---

## 설계가 먼저인 이유

많은 분들이 "일단 만들어보자"고 시작합니다. 그러면 어떻게 될까요?

```
설계 없이 바로 만들기:
  30분 헤맴 → 결과물이 애매함 → 수정 반복 → 지침

설계 먼저:
  10분 설계 → 5분 구현 → 바로 써먹음
```

코딩보다 중요한 건 **"무엇을 만들지"를 명확하게 정하는 것**입니다.

잘 쓴 설계서 = 잘 만든 스킬. **설계서가 곧 스킬의 절반입니다.**

---

## 생각해봅시다

::: info 잠깐, 눈을 감고 생각해보세요 (30초)
"내 업무 중에서 가장 귀찮은 반복 작업은 뭔가요?"

매주 반복되는 것, 할 때마다 힘 빠지는 것, 형식은 정해져 있는데 매번 새로 써야 하는 것...
:::

::: details 힌트: 이런 것들이 스킬로 만들기 좋아요
- 매주 같은 형식으로 쓰는 보고서
- 이메일 답장할 때마다 나오는 비슷한 패턴
- 회의 후 정리해야 하는 회의록
- SNS에 올릴 게시글 캡션 쓰기
- 데이터를 정해진 형식으로 변환하기
- 고객 문의에 대한 답변 초안
- 기획서나 제안서의 특정 섹션 작성
- 주간/월간 리포트 요약

"이거 매번 쓰기 귀찮은데..."라는 생각이 드는 게 있다면, 그게 바로 스킬 후보입니다.
:::

## 스킬 설계서 템플릿

::: tip 이 종이 한 장이면 스킬 완성의 절반입니다
아래 템플릿을 종이에 적거나 메모 앱에 옮겨서 채워보세요.

**빈칸을 전부 채울 필요 없습니다.** 핵심만 적으면 됩니다.
:::

```markdown
# 나의 스킬 설계서

## 1. 선언
- 스킬 이름:
- 한 줄 설명:
- MVP 목표: "____하면 → ____를 만들어준다"

## 2. 언제 쓰나요?
- 대표 상황:
- 왜 필요한가:

## 3. 사용법
- 이렇게 부르면: "/스킬이름 ____" 또는 "____해줘"
- 결과물:

## 4. 범위
- 하는 것 (3개 이내):
  1.
  2.
  3.
- 안 하는 것 (2개 이내):
  1.
  2.

## 5. 완료 기준
"이렇게 되면 성공":
```

## 작성 가이드

각 항목을 어떻게 채울지 설명합니다.

### 1. 선언 — 가장 중요!

**스킬 이름**은 짧고 직관적으로 짓습니다. 부를 때 쓰는 이름입니다.

> 좋은 예: `주간보고`, `고객답변`, `SNS캡션`, `회의록`
> 피할 것: `내업무자동화스킬v2`, `만능AI도구`

**MVP 목표**는 "트리거 → 결과" 한 문장입니다.

> 예: "주간 업무 내용을 입력하면 → 보고서 형식으로 정리해준다"
> 예: "고객 문의 내용을 붙이면 → 친절한 답변 초안을 만들어준다"

::: tip 💡 꿀팁
이름 짓기가 어려우면 "내가 이 스킬을 부를 때 뭐라고 부를까?"를 먼저 생각해보세요. 자연스럽게 입에서 나오는 단어가 좋은 이름입니다.
:::

### 2. 언제 쓰나요?

구체적인 상황 **하나**만 적으세요. 많으면 범위가 흐릿해집니다.

> 좋은 예: "매주 금요일 오후, 팀장님께 주간 업무 보고할 때"
> 피할 것: "보고서 쓸 때 전반적으로"

::: tip 💡 꿀팁
"언제"가 구체적일수록 스킬이 더 정확해집니다. "매주 금요일 3시"처럼 시간까지 떠올릴 수 있다면 충분히 구체적인 거예요.
:::

### 3. 사용법

실제로 어떻게 부를지 상상해보세요. 그리고 결과물이 어떤 형태인지 정합니다.

> 예: `/주간보고 이번 주 한 일: ...` → 보고서 형식 텍스트
> 예: `이 문의에 답변 써줘: [문의 내용]` → 답변 이메일 초안

::: tip 💡 꿀팁
"내가 지금 당장 쓴다면 어떻게 입력할까?"를 상상해보세요. 그게 바로 사용법입니다. 너무 어렵게 생각하지 않아도 됩니다.
:::

### 4. 범위 — "안 하는 것"이 더 중요!

범위를 좁힐수록 스킬이 정확해집니다. "안 하는 것" 두 가지를 먼저 정하면 "하는 것"이 자연스럽게 좁혀집니다.

> 예 (SNS캡션 스킬):
> - 안 하는 것: 이미지 생성, 해시태그 리서치
> - 하는 것: 캡션 텍스트만 작성, 이모지 포함

::: tip 💡 꿀팁
"이건 안 해도 돼"를 먼저 적어보세요. 뭔가를 빼면 나머지가 선명해집니다. 처음 스킬은 작을수록 좋습니다.
:::

### 5. 완료 기준

"이게 나오면 성공" 한 문장. 나중에 테스트할 때 기준이 됩니다.

> 예: "내가 쓴 내용과 거의 비슷한 퀄리티의 보고서가 3분 안에 나온다"
> 예: "복사해서 바로 보낼 수 있는 수준의 답변이 나온다"

::: tip 💡 꿀팁
"복사해서 바로 쓸 수 있다"는 기준이 가장 좋습니다. 그게 스킬의 존재 이유니까요.
:::

## 발표 & 피드백 (10분)

설계서가 완성되면 2-3분씩 발표합니다.

**발표 포맷** (이 세 가지만 말하면 됩니다)

```
1. 스킬 이름
2. MVP 목표 한 문장
3. "이렇게 되면 성공" 한 문장
```

**피드백 방식**
- "이 부분이 좋다: ___"
- "이렇게 하면 더 좋겠다: ___"

::: info 정답은 없습니다
발표가 부담스러울 수 있지만, 지금 이 자리는 완성도를 평가하는 곳이 아닙니다. 비판이 아니라 같이 다듬는 시간입니다. 어떤 설계서든 이야기할 거리가 있고, 누군가의 피드백이 아이디어를 더 좋게 만들어줍니다. 편하게 꺼내주세요.
:::

## 설계서 → 실제 스킬로 만들기

::: info 다음 단계
설계서가 완성되면, 아까 만든 이메일 초안 스킬처럼 SKILL.md로 변환할 수 있습니다.

**방법**은 간단합니다. Claude Code에게 이렇게 말하면 됩니다:

```
이 설계서를 바탕으로 SKILL.md 파일을 만들어줘.

[설계서 내용 붙여넣기]
```

AI가 알아서 SKILL.md 형식으로 변환해줍니다!

오늘 시간이 되면 함께 해보고, 안 되면 집에서 혼자 해볼 수 있습니다.
예시 설계서와 완성된 SKILL.md는 [참고자료 → 설계서 예시](/학습자료/4-실습/4-2-설계서예시)에서 볼 수 있습니다.

처음부터 완벽할 필요 없습니다. 오늘 하나 만든 것만으로도 충분히 잘 하셨습니다. 앞으로 하나씩 늘려가다 보면, 어느 날 "나는 AI를 제대로 쓸 줄 아는 사람"이 되어 있을 거예요. 오늘 그 첫 번째 스킬이 탄생했습니다. 정말 수고하셨습니다!
